Lightweight time-series forecasting for server-load prediction & SLO-constrained rightsizing

Forecast load on CPUs alone — no GPU — and compute the smallest resource allocation that preserves the SLO via integer programming. It monitors and resizes real GCP instances.

The problem

On-premises chronically over-provisions resources

  • Azure production trace (SOSP'17): ~60% of VMs average under 20% CPU. Most allocated capacity sits idle, burning cost and power.
  • Public-cloud auto-optimizers (CAST AI, AWS Compute Optimizer, …) cannot run in air-gapped / regulated on-prem — they require sending telemetry off-site.
  • On-prem monitoring tools stop at threshold-based reporting — they do not solve "how much can we safely cut?" as constrained optimization.

The gap

No tool computes, with no external dependency and no GPU, the smallest resources that won't break the SLO.

MetricLens fills exactly this gap.

Sources: Cortez et al., "Resource Central", SOSP 2017 · Barroso & Hölzle, "The Datacenter as a Computer".

One question drives everything — "how much can we cut without breaking the SLO?" The next slides explain each stage's concept in turn.
Concept ① · What & why we forecast

Time-series forecasting: estimate the future from past observations

  • Input: per-host hourly time series of CPU·memory·network — an ordered sequence y₁, y₂, …, yₙ.
  • Output: the value h steps ahead ŷ(n+h) and its uncertainty (a prediction interval).
  • Server-load series carry three properties at once: trend (slow rise/fall), seasonality (24h daily cycle), autocorrelation (the recent state influences the next).

→ Modelling each of these separately is the key to accuracy.

Design constraints

  • Standard library only — no native deps, no GPU
  • White-box — every step is verifiable and explainable
  • Lightweight — minimal container footprint and inference cost
Concept ② · Seasonal-trend decomposition

Split the signal into trend + seasonal + residual

y[t] = trend[t] + seasonal[t mod p] + residual[t]
  • Trend: OLS line over the sample index. Fitted on per-period block means so seasonality cannot leak into the slope.
  • Seasonal: per-phase (t mod p) mean of the detrended series, re-centred to sum to zero so it does not bias the trend.
  • Residual: what trend and seasonal do not explain. Its spread sets the prediction-interval width.
Concept ③ · AR(1) residual correction

Carry the recent state forward to sharpen the forecast

Residuals of real CPU signals are not random — they are autocorrelated. A large residual now makes the next one likely large too.

  • Estimate the residuals' lag-1 autocorrelation ρ (clamped to 0–0.95).
  • Add ρh × the latest residual to the decomposition forecast — the influence decays geometrically with horizon h.
  • On real CPU traces ρ is often 0.5–0.9, so this term yields a large gain.
ŷ = trend + seasonal + ρh · residual[latest]

This term lets the model beat both last-value and seasonal-naive baselines.

Concept ④ · Uncertainty (prediction interval)

A point forecast is not enough — quantify uncertainty as an interval

  • Resizing is a risky decision. Cutting on a single point forecast ŷ violates the SLO whenever the forecast is wrong.
  • So we also emit a 95% prediction interval:
[lower, upper] = ŷ ± 1.96 · RMSE
  • RMSE comes from an out-of-sample backtest — expanding-window 1-step forecasts compared to actuals. It reflects true predictive uncertainty.
  • With too little history, it falls back to the decomposition residual's standard deviation.

Is the interval well-calibrated? — PICP

PICP (prediction-interval coverage probability) = fraction of actuals that land inside the interval. It should sit near the nominal 0.95.

0.93–0.98measured PICP (nominal 0.95)

→ Uncertainty is neither over- nor under-stated: well calibrated.

Concept ⑤ · Robust peak (p95)

Size to the p95, not the maximum

Rightsizing is driven by "peak load". But sizing to a single spike forces permanent over-provisioning for that one event.

  • So we use the 95th percentile (p95) as the peak (nearest-rank), not the max.
  • It covers nearly all of the load while staying robust to rare transient bursts.
  • This is the standard robust statistic for capacity sizing.
Right: the lone burst (max=98) is ignored; p95=64 is adopted as the peak.
Concept ⑥ · SLO-constrained integer program

Solve exactly for the smallest integer allocation that keeps the SLO

SLO (Service-Level Objective): the target the service must meet (e.g. 99.9% availability). Here the SLO holds when utilisation stays under target even at the peak.

Optimization — minimize cost (= allocation) subject to the headroom constraint:

peakp95 · safety_margin ≤ target_util · allocation
  • vCPU is an integer, memory is in 256 MB blocks — the allocatable units are integers, so this is an integer program.
  • The search space is small and bounded, so exact enumeration guarantees the global optimum — no external solver.

Why integer programming

  • Rounding up a continuous-relaxation solution can subtly violate the SLO.
  • Finding the smallest integer that satisfies the constraint guarantees minimum cost with no violation.
  • If the constraint is infeasible (already under-provisioned), keep the current allocation — a safe fallback.
Concept ⑦ · Safety margin & target utilisation (worked example)

What the parameters mean, and the actual computation

  • target_util = 0.65: steady-state utilisation ceiling. The other 35% is headroom for bursts and error.
  • safety_margin = 1.2: multiplicative buffer for forecast error — assumes 20% more load than predicted.
  • Together they hold the target SLO confidence (e.g. 99.9%).

→ More conservative (lower target, higher margin) is safer but saves less. Tunable by operational policy.

Example — a 4-vCPU host

Forecast peak CPU35%
Load (in vCPU)0.35 × 4 × 1.2 = 1.68
Constraint1.68 ≤ 0.65 × n
Smallest integer nn ≥ 2.58 → 3 vCPU
Result4 → 3 vCPU (−25%)

Memory follows the same procedure in 256 MB blocks.
