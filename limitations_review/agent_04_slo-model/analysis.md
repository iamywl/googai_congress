## 0. 관점 메모
이 분석 렌즈는 MetricLens AI 프로젝트에서 SLO(Service Level Objective) 모델링 방식의 한계에 집중한다. 특히 `target_utilisation`, `safety_margin`, `slo_confidence`와 같은 핵심 파라미터들이 정적인 상수로 설정되어 있으며, 이러한 설정이 실제 서비스 가용성이나 지연 시간과 같은 운영 SLO 지표와 어떻게 연결되는지에 대한 근거가 부족하다는 점을 심층적으로 검토한다.

## 1. 핵심 한계 요약
*   `target_utilisation`, `safety_margin`, `slo_confidence` 등 SLO 관련 핵심 파라미터가 정적인 상수로 설정되어 현실의 동적인 SLO 요구사항을 반영하지 못한다.
*   "피크 이용률 ≤ 목표"라는 리소스 활용률 기반의 제약 조건이 실제 서비스 가용성(availability)이나 지연 시간(latency) 등 사용자 체감 SLA를 직접 보장한다는 정량적 근거가 부재하다.
*   큐잉 이론, 대기 시간, 테일 레이턴시 등 시스템 부하에 따른 서비스 품질 변화를 모델링하는 요소가 완전히 누락되어 있다.
*   `slo_confidence` 값이 예측 모델의 불확실성이나 실제 서비스 영향과 독립적으로 전달되며, 계산에 활용되지 않아 "SLO 보장" 주장의 정량적 연결고리가 약하다.

## 2. 상세 분석

### 2.1 SLO 관련 파라미터의 정적 상수화
*   **심각도**: 높음
*   **무엇이**: 시스템의 핵심 SLO 관련 파라미터인 `target_utilisation`, `safety_margin`, `slo_confidence`가 `backend/app/config.py`에 다음과 같이 정적인 상수로 정의되어 있다:
    ```python
    target_utilisation: float = 0.65     # steady-state utilisation ceiling
    safety_margin: float = 1.2           # forecast-error buffer
    slo_confidence: float = 99.9         # SLO availability the margin protects
    ```
*   **근거**: `backend/app/config.py` 파일의 `Settings` 클래스 정의. `backend/app/core/optimizer.py`의 `recommend_resize` 함수에서도 이 값들을 직접 사용하며 `slo_confidence`는 최종 `ResizeRecommendation` 객체에 그대로 전달된다.
    ```python
    # backend/app/core/optimizer.py
    def recommend_resize(...
        target_utilisation: float = 0.65,
        safety_margin: float = 1.2,
        slo_confidence: float = 99.9,
    ) -> ResizeRecommendation:
        ...
        slo_confidence=round(slo_confidence, 2),
    ```
*   **왜 문제**: 실제 운영 환경에서 SLO는 서비스의 중요도, 워크로드 특성, 비용 제약 등에 따라 유동적으로 변화해야 한다. 정적인 상수는 이러한 동적인 요구사항을 반영할 수 없으며, 과도한 자원 낭비(지나치게 보수적인 값) 또는 SLO 위반(지나치게 공격적인 값)으로 이어질 수 있다. 특히 `slo_confidence=99.9`는 보편적인 고가용성 목표치일 뿐, 이 값이 어떻게 `target_utilisation`이나 `safety_margin`과 상호작용하여 이룩되는지 설명되지 않는다.
*   **어떤 조건에서 드러나는가**: 다양한 서비스 중요도를 가진 애플리케이션이나, 계절성 또는 이벤트성 부하로 인해 SLO 목표가 변경되는 상황에서 이러한 정적 설정은 최적의 자원 관리를 방해한다.

### 2.2 리소스 활용률 기반 제약의 SLO 보장 근거 취약성
*   **심각도**: 치명
*   **무엇이**: 최적화 엔진의 핵심 제약 조건은 `projected_peak_load * safety_margin <= target_utilisation * allocation` 이다. 즉, 예측된 피크 부하(안전 마진 적용)가 할당된 자원의 목표 활용률을 초과하지 않도록 하는 것이다. 이는 `backend/app/core/optimizer.py`의 `_smallest_unit_allocation` 함수에서 `load_units <= target_utilisation * units` 형태로 구현된다.
*   **근거**: `backend/app/core/optimizer.py`의 `recommend_resize` 및 `_smallest_unit_allocation` 함수 구현. `docs/01_functional_spec.md`의 `US-2` ("A 서버 vCPU를 50% 축소해도 99.9% SLO 유지 가능"과 같은 정량 근거를 받아 자원 재배치를 결정하고 싶다) 및 `US-3` (p95 기반의 안정적 권장을 받고 싶다)는 SLO를 강조하지만, 이를 리소스 활용률로만 접근한다. `docs/08_competitive_analysis.md`에서도 "SLO 제약 정수계획 기반 처방적 최적화"를 차별점으로 내세우지만, 이 제약의 실질적인 의미가 리소스 활용률 제약에 머무른다.
*   **왜 문제**: CPU나 메모리 사용률이 특정 임계값 이하로 유지된다고 해서 반드시 서비스의 가용성(예: HTTP 5xx 에러율)이나 지연 시간(예: p99 응답 시간)이 보장되는 것은 아니다. 예를 들어, CPU 사용률이 낮아도 I/O 병목이나 컨텐션으로 인해 응답 시간이 급증할 수 있으며, 서비스 장애가 발생할 수 있다. "피크 이용률 ≤ 목표"는 시스템의 건강 상태를 나타내는 지표 중 하나일 뿐, 사용자 경험이나 비즈니스 목표와 직접적으로 연관되는 SLO와는 괴리가 존재한다.
*   **어떤 조건에서 드러나는가**: I/O 바운드 워크로드, 네트워크 지연, 데이터베이스 병목, 컨텐션이 심한 공유 자원 환경, 또는 특정 비즈니스 로직 처리 시간이 긴 경우, 단순히 CPU/메모리 활용률만을 제어하는 방식으로는 SLO를 보장하기 어렵다.

### 2.3 큐잉 이론, 대기 시간, 테일 레이턴시 모델링의 부재
*   **심각도**: 치명
*   **무엇이**: 현재 SLO 모델링은 시스템 부하가 증가함에 따라 발생하는 요청 큐잉, 처리 지연, 그리고 특히 사용자 경험에 큰 영향을 미치는 테일 레이턴시(p99, p99.9 응답 시간)에 대한 명시적인 모델링 또는 고려 사항이 전혀 없다.
*   **근거**: `backend/app/core/optimizer.py` 및 `backend/app/core/forecaster.py`를 포함한 전체 코드베이스에서 큐잉 모델, Little's Law, 응답 시간 분포, 테일 레이턴시 예측과 관련된 로직이나 파라미터가 발견되지 않는다. `docs/04_database_design.md`의 `metrics` 테이블 스키마에는 `cpu_pct`, `mem_pct`, `net_in_kbps`, `net_out_kbps`만 포함되어 있으며, 지연 시간(latency) 관련 메트릭은 없다.
*   **왜 문제**: 대부분의 사용자 중심 SLO는 응답 시간이나 처리량, 에러율과 같이 직접적으로 사용자 경험에 영향을 미치는 지표를 기반으로 한다. 자원 활용률이 낮더라도, 특정 임계값을 넘어서는 순간 대기 시간이 기하급수적으로 증가하거나(큐잉 이론), 일부 요청의 응답 시간이 극단적으로 길어지는(테일 레이턴시) 현상은 흔히 발생한다. 이러한 현상을 모델링하지 않으면, MetricLens의 "SLO 보장" 주장은 현실적인 서비스 품질 보장과 거리가 멀어진다.
*   **어떤 조건에서 드러나는가**: 트래픽이 급증하는 상황, 마이크로서비스 간의 복잡한 의존성으로 인한 분산 트레이싱, 그리고 엄격한 지연 시간 목표를 가진 실시간 시스템(예: 금융 거래, 게임 서버)에서 이러한 한계는 치명적인 SLO 위반으로 이어진다.

## 3. 암묵적 가정과 그 취약성
현재 설계는 다음과 같은 암묵적 가정을 하고 있으며, 이들이 깨질 경우 SLO 보장 주장은 무력해진다:

*   **가정**: 자원 활용률(CPU, Memory)과 서비스 품질(Availability, Latency) 간에 선형적이고 직접적인 관계가 존재한다. 즉, 활용률이 낮으면 서비스 품질은 항상 좋다.
*   **취약성**: 앞서 언급했듯이, I/O 병목, 데드락, 락 컨텐션, GC(Garbage Collection) 지연, 네트워크 포화 등 다양한 요인이 자원 활용률과 독립적으로 서비스 품질에 악영향을 미칠 수 있다. 특정 자원의 활용률만으로 전체 시스템의 건강을 판단하는 것은 위험하다.
*   **가정**: `safety_margin=1.2`와 `slo_confidence=99.9%`는 모든 워크로드와 SLO 시나리오에 보편적으로 적용 가능한 값이다.
*   **취약성**: 워크로드의 변동성, 예측 모델의 오차 특성, 그리고 서비스의 중요도에 따라 필요한 안전 마진은 크게 달라진다. 99.9% 가용성을 달성하기 위한 자원 보증은 99% 가용성보다 훨씬 높은 마진을 요구할 수 있으며, 이를 단일 상수로 일반화하는 것은 비현실적이다.

## 4. 파급 효과
*   **오버/언더 프로비저닝 위험 증가**: 정적 SLO 파라미터는 실제 SLO 목표에 비해 자원을 과도하게 할당하거나(비용 낭비), 반대로 부족하게 할당하여 서비스 중단을 초래할 수 있다.
*   **시스템 신뢰도 저하**: "SLO 보장"이라는 MetricLens의 핵심 가치가 실제 운영에서 SLO 위반으로 이어질 경우, 시스템에 대한 운영자의 신뢰도가 크게 하락하고 도입 동력을 잃을 수 있다.
*   **잘못된 의사결정**: 활용률 지표만을 기반으로 한 리사이징 권고는 운영자가 실제 서비스 품질 문제를 간과하고, 잘못된 자원 배분 결정을 내리게 유도할 수 있다.
*   **예측 가능한 서비스 장애**: 테일 레이턴시나 큐잉 효과를 모델링하지 않으면, 피크 트래픽 시나리오에서 서비스가 갑작스럽게 성능 저하를 겪거나 장애로 전환될 위험이 높아진다.

## 5. 개선 제안
1.  **동적인 SLO 파라미터 구성**: `target_utilisation`, `safety_margin`, `slo_confidence`를 호스트별 또는 워크로드별로 구성 가능하도록 변경한다. 예를 들어, API를 통해 동적으로 설정하거나, 설정 파일을 통해 관리하도록 한다.
2.  **예측 구간 기반의 안전 마진 동적 산출**: `docs/12_limitations.md`의 "2.3 "SLO 보장"의 정량적 연결고리 약함"에 제시된 대로, `safety_margin`을 예측 모델의 신뢰 구간(Prediction Interval)이나 불확실성 추정치에서 동적으로 유도하여 SLO 위반 확률을 명시적으로 계산하도록 개선한다.
3.  **지연 시간(Latency) 메트릭 통합**: `metrics` 테이블에 응답 시간(p99, p99.9 레이턴시) 또는 큐 길이 등의 서비스 품질 지표를 추가하고, 이를 예측 및 최적화 모델에 반영한다. 이를 통해 자원 활용률뿐만 아니라 실제 서비스 품질 지표를 SLO 제약 조건으로 활용한다.
4.  **큐잉 이론 모델 통합**: `optimizer.py`에 큐잉 이론(예: M/M/1 모델)을 통합하여 자원 활용률과 예상 대기 시간/응답 시간 간의 관계를 모델링한다. 이를 통해 특정 활용률에서 서비스 지연이 어떻게 변하는지 예측하고, 이를 최적화에 반영한다.
5.  **다중 SLO 목표 최적화**: 단일 목표(비용 최소화)를 넘어, 비용 최소화와 성능 목표(최대 지연 시간, 최소 처리량) 동시 달성을 고려하는 다중 목표 최적화 기법을 도입한다.

## 6. 타 관점과의 연결
*   **예측 모델의 표현력 한계 (#01)**: 예측 모델이 비선형성, 다중 계절성, 다변량 상호작용을 포착하지 못하면, SLO 보장을 위한 정확한 부하 예측 자체가 불가능해진다. 예측 오차가 클수록 `safety_margin`이 무의미해진다.
*   **예측 불확실성 정량화의 한계 (#02)**: 예측 구간의 보정도가 낮거나 정규성/등분산 가정이 현실과 다르면, 이를 기반으로 산출되는 `safety_margin`은 실제 SLO를 보장하지 못하게 된다.
*   **최적화의 한계 (#03)**: 현재의 호스트별 독립 최적화 방식으로는 플릿 전체의 SLO 목표를 달성하기 위한 통합적인 자원 재배치(예: 저활용 호스트의 통합)가 불가능하다. 이는 플릿 수준의 SLO 관리에 큰 한계로 작용한다.
*   **비용 모델의 한계 (#09)**: 실제 SLO 위반으로 인한 기회비용(예: 비즈니스 손실)이 비용 모델에 반영되지 않으면, 리사이징 결정의 경제적 타당성이 왜곡될 수 있다.
*   **설명가능성·운영 UX의 한계 (#14)**: 현재의 "SLO 신뢰도"가 정량적 근거 없이 전달되면, 운영자는 이 수치를 신뢰하기 어렵고, SLO 위반 시 책임 소재를 파악하기 어려워 시스템 채택률을 저해할 수 있다.
