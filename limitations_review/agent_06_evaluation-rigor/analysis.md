## 0. 관점 메모
이 분석 렌즈는 MetricLens AI 프로젝트의 모델 평가 방식과 결과의 엄밀성(rigor)에 집중하여 한계점을 탐색합니다. 특히 예측 모델의 성능 검증이 단일 스텝 백테스트에 국한되는지, 시험 데이터의 대표성이 충분한지, 그리고 리사이징 '결정'의 실제적인 비즈니스 가치(비용 절감, SLO 유지)가 엔드투엔드(end-to-end)로 평가되는지를 중점적으로 살펴봅니다.

## 1. 핵심 한계 요약
*   **예측 모델 평가가 1-스텝 백테스트에만 국한**: 장기 예측 정확도 및 신뢰구간 평가가 부재하여 실제 리사이징 결정에 필요한 미래 구간의 예측 성능이 검증되지 않음.
*   **자기 생성 데이터에 대한 자기 평가의 순환성**: 모델이 유리하도록 설계된 합성 데이터셋으로 모델을 평가하여 외부 타당성과 일반화 가능성이 현저히 낮음.
*   **리사이징 결정의 엔드투엔드 평가 부재**: 예측 정확도는 평가되나, 예측을 기반으로 한 리사이징 권고가 실제 비용 절감 및 SLO 유지에 미치는 영향(다운타임, 플래핑 등 운영 현실 반영)은 평가되지 않음.
*   **제한적인 워크로드 아키타입과 기준선**: 6가지 합성 아키타입만으로 평가하며, naive 및 seasonal-naive 외에 다른 강력한 기준선 모델과의 비교가 부족함.

## 2. 상세 분석

*   **1-스텝 확장 윈도우 백테스트에만 국한**
    *   **심각도**: 높음
    *   **무엇이 / 근거(인용)**: `docs/11_model_evaluation.md`의 "방법론" 섹션에서 "확장 윈도우 **1-스텝 백테스트**로 모델을 두 표준 기준선과 비교한다"고 명시되어 있으며, `backend/app/core/evaluation.py`의 `backtest_model` 함수 또한 `horizon=1`로 고정되어 있습니다. `docs/12_limitations.md` §1.2 "1-스텝(horizon=1)만 검증됨"에서 "h>1 예측의 정확도 근거가 없다"고 스스로 인정합니다.
    *   **왜 문제**: 서버 리소스 리사이징은 대개 며칠 또는 몇 주 단위의 장기적인 자원 계획에 기반합니다. 1-스텝(60분) 예측의 정확도만으로 시스템의 장기적인 리사이징 결정의 유효성을 보장하기 어렵습니다. 시간이 지남에 따라 예측 오차는 누적되고 신뢰구간은 확장되는데, 이에 대한 평가가 없어 실제 운영 환경에서의 예측 신뢰도를 과대평가할 수 있습니다.
    *   **어떤 조건에서 드러나는가**: 리사이징 정책이 단기적인 부하 변동(예: 1시간)이 아닌, 1일/1주/1개월과 같이 더 긴 시간의 부하 패턴에 기반해야 할 때, 혹은 예측을 이용해 장기적인 용량 계획을 수립해야 할 때 이 한계가 명확하게 드러납니다.

*   **자기 생성 데이터에 대한 자기 평가의 순환성**
    *   **심각도**: 치명
    *   **무엇이 / 근거(인용)**: `docs/12_limitations.md` §3.1 "자기 생성 데이터로 자기 모델 평가"에서 "평가는 [`workload.py`](../backend/app/core/workload.py)가 만든 합성 데이터에서만 수행된다. ... 모델이 유리하도록 데이터 생성 과정에 정렬된 **순환 구조**다."라고 설명합니다. `backend/app/core/workload.py`에서는 AR(1) 잡음을 주입하고, `backend/app/core/forecaster.py`는 AR(1) 잔차 보정을 모델에 적용합니다.
    *   **왜 문제**: 모델이 특정 통계적 특성을 가지도록 생성된 데이터셋으로 해당 모델의 성능을 평가하는 것은 "데이터 피팅"에 가깝습니다. 이는 외부 데이터, 즉 실제 운영 환경의 불규칙하고 예측 불가능한 부하 패턴에 대한 모델의 일반화(generalization) 성능을 과도하게 낙관적으로 평가하게 만듭니다. `docs/11_model_evaluation.md`의 "MASE<1, DM p<0.05로 유의하게 우수"와 같은 결과가 이 순환 구조 위에 있어 외적 타당성이 제한적입니다.
    *   **어떤 조건에서 드러나는가**: MetricLens AI가 실제 데이터센터 트레이스나 새로운 종류의 워크로드에 적용될 때, 문서에 제시된 것보다 훨씬 낮은 예측 정확도와 SLO 유지율을 보일 가능성이 높습니다.

*   **리사이징 '결정'의 엔드투엔드 비용·SLO 위반 평가 부재**
    *   **심각도**: 높음
    *   **무엇이 / 근거(인용)**: `docs/11_model_evaluation.md`의 모든 지표(`MAPE`, `RMSE`, `MASE`, `PICP`, `DM stat`)는 예측 모델의 성능에 집중하며, 리사이징 권고(`backend/app/core/optimizer.py`)가 실제 시스템 운영(비용, SLO)에 미치는 영향에 대한 엔드투엔드 시뮬레이션이나 평가 지표가 없습니다. `docs/12_limitations.md` §2.3 "SLO 보장"의 정량적 연결고리 약함" 및 §2.4 "비용·운영 현실 미반영"에서 이 문제가 지적됩니다.
    *   **왜 문제**: MetricLens의 핵심 목표는 "SLO를 보장하는 최소 자원 구성을 산출"하는 것이므로, 단순히 예측이 정확하다는 것만으로는 충분하지 않습니다. 예측 오차가 리사이징 결정에 어떤 영향을 미치고, 이 결정이 궁극적으로 SLO 위반으로 이어지는지, 또는 불필요한 비용을 발생시키는지에 대한 평가가 없습니다. `est_cost_saving_pct`는 vCPU/메모리 감소율의 단순 평균이며 실제 단가와 연동되지 않습니다.
    *   **어떤 조건에서 드러나는가**: 리사이징 권고가 적용된 후 시스템의 실제 비용 절감 효과가 기대에 미치지 못하거나, 예상치 못한 SLO 위반이 발생할 때 이 평가의 공백이 심각한 문제로 대두됩니다.

*   **제한적인 워크로드 아키타입과 기준선**
    *   **심각도**: 중간
    *   **무엇이 / 근거(인용)**: `docs/09_workload_modeling.md`와 `docs/11_model_evaluation.md`에 따르면 평가는 "6 아키타입"에 대해서만 수행됩니다. 기준선 모델은 `backend/app/core/evaluation.py`에 정의된 `backtest_naive` (직전값)와 `backtest_seasonal_naive` (계절-직전값)뿐입니다.
    *   **왜 문제**: 6가지 아키타입은 실제 데이터센터의 복잡성을 완전히 대변하기에는 부족하며, 특히 예측하기 어려운 이상 패턴(블랙스완 이벤트, 급격한 트래픽 변화 등)에 대한 모델의 강건성을 검증하기 어렵습니다. 또한, naive한 기준선 외에 더 진보된 통계 모델(예: FB Prophet, ARIMA, Holt-Winters 등)과의 비교가 없어 MetricLens 모델의 상대적인 우수성을 객관적으로 평가하기 어렵습니다.
    *   **어떤 조건에서 드러나는가**: MetricLens가 6가지 아키타입과 통계적 특성이 다른 새로운 워크로드에 적용되거나, 모델 성능의 객관적인 비교가 필요한 경우 이 한계가 부각됩니다.

## 3. 암묵적 가정과 그 취약성
*   **가정 1: 1-스텝 예측 정확도가 장기 예측 정확도를 대변한다.**
    *   **취약성**: 시계열 예측에서 예측 지평이 길어질수록 불확실성은 증가하고 정확도는 감소하는 것이 일반적입니다. 1-스텝 예측이 충분히 정확하다고 해서 리사이징 결정에 필요한 더 긴 지평의 예측도 유효하다는 보장이 없습니다.
*   **가정 2: 합성 데이터셋의 통계적 대표성은 실제 환경에서도 유지된다.**
    *   **취약성**: `docs/09_workload_modeling.md`에서 "실측 데이터센터 통계에 근거(calibration)"했다고 주장하지만, 합성 데이터는 실제 데이터의 미묘한 패턴, 예외 상황, 갑작스러운 변화를 완벽하게 재현하기 어렵습니다. 특히 모델이 생성 데이터의 특정 통계적 특성(예: AR(1) 잡음)에 최적화되어 있다면, 이 가정이 깨질 경우 모델의 성능은 급격히 저하될 것입니다.
*   **가정 3: 예측 모델의 통계적 우월성(RMSE, MASE 등)이 곧 비즈니스 가치(비용 절감, SLO 유지)로 직결된다.**
    *   **취약성**: 예측 모델의 통계적 오차 지표가 낮다고 해서 그 예측이 리사이징 결정에 항상 최적의 결과를 가져오는 것은 아닙니다. 예측 오차의 방향(과대/과소 예측), 분포, 그리고 리사이징 정책(안전 마진, 목표 가동률)과의 상호작용이 실제 비용과 SLO에 더 큰 영향을 미칠 수 있습니다.

## 4. 파급 효과
*   **예측 기반 리사이징 결정의 신뢰도 저하**: 1-스텝 예측에만 의존하고 합성 데이터로 검증된 모델은 실제 운영 환경에서 예상치 못한 과/저 프로비저닝을 유발하여 시스템 안정성 저해 및 비용 증가로 이어질 수 있습니다.
*   **비즈니스 가치 평가의 불확실성**: 리사이징 권고가 실제 비용 절감이나 SLO 유지에 미치는 영향에 대한 엔드투엔드 평가가 없기 때문에, MetricLens AI의 핵심 주장인 "OPEX·전력 낭비 절감" 및 "SLO 보장"이 정량적으로 입증되지 않아, 시스템 도입에 대한 의사결정을 어렵게 만듭니다.
*   **운영 리스크 증가**: 예측 오차의 누적 효과나 리사이징 결정의 비선형적 결과에 대한 평가가 없어, 실제 리사이징 적용 시 예측치 못한 다운타임, 성능 저하, 또는 잦은 리사이징(플래핑)으로 인한 운영 복잡성 증가 등의 리스크를 관리하기 어렵습니다.

## 5. 개선 제안
*   **다중 스텝 예측 백테스트 및 평가 지표 추가**: 1-스텝 외에 6시간, 12시간, 24시간, 7일 등 다양한 예측 지평에 대한 모델 성능(`mape`, `rmse`, `coverage` 등)을 `evaluation.py`에 추가하고 `docs/11_model_evaluation.md`에 보고합니다. (영향: 높음)
*   **실제 데이터셋 기반 모델 재평가**: `docs/12_limitations.md` §3.1에서 제안된 바와 같이, Azure/Alibaba 등 **실제 데이터셋(홀드아웃)**으로 모델을 재평가하여 외부 타당성을 확보합니다. 이는 모델의 일반화 성능을 객관적으로 입증하는 데 필수적입니다. (영향: 치명)
*   **리사이징 결정의 엔드투엔드 시뮬레이션 및 비용/SLO 평가**: 예측 모델과 최적화 엔진을 결합하여, 리사이징 권고가 시간 경과에 따라 가상 환경에서 실제 비용 절감 및 SLO 유지에 어떤 영향을 미치는지 시뮬레이션하고 평가하는 프레임워크를 구축합니다. 여기에는 다운타임, 플래핑, 실제 머신 타입 단가 등을 반영해야 합니다. (영향: 높음)
*   **다양한 기준선 모델과의 비교**: naive 및 seasonal-naive 외에, Holt-Winters, ARIMA, 또는 간단한 머신러닝 모델(선형 회귀, 트리 기반 모델 등)을 기준선으로 추가하여 MetricLens 모델의 상대적인 성능 우위를 보다 엄격하게 검증합니다. (영향: 중간)

## 6. 타 관점과의 연결
*   **예측 모델 (01|forecasting-model)**: 다중 스텝 예측 검증은 예측 모델 자체의 "표현력 한계"와도 밀접하게 연결됩니다. 장기 예측이 불확실하다면 이는 모델 표현력의 근본적인 한계일 수 있습니다.
*   **불확실성 정량화 (02|uncertainty)**: 다중 스텝 예측에 대한 평가가 이루어지면 예측 지평에 따른 불확실성 증가가 명확히 드러나고, 이를 통해 예측 구간의 보정 방식에 대한 "불확실성 정량화의 한계"를 더 깊이 분석할 수 있습니다.
*   **최적화 (03|optimization)**: 리사이징 결정의 엔드투엔드 평가 부재는 최적화 엔진이 도출하는 "최적해"가 실제 비즈니스 목표와 얼마나 부합하는지에 대한 검증의 공백으로 이어집니다.
*   **SLO 모델링 (04|slo-model)**: 리사이징 결정의 SLO 위반 평가 부재는 "SLO 모델링의 한계"와 직접적으로 연결되며, SLO 보장 주장의 정량적 근거를 약화시킵니다.
*   **데이터 대표성 (05|data-representativeness)**: 자기 생성 데이터에 대한 자기 평가는 "데이터 대표성의 한계"와 직결되는 가장 중요한 부분입니다. 실제 데이터를 사용한 평가는 이 한계를 해소하는 핵심적인 단계입니다.
