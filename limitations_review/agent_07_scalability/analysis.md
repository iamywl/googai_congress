## 0. 관점 메모
이 분석은 MetricLens AI의 확장성 및 플릿 규모 처리 능력에 초점을 맞춥니다. 특히 수천 대의 호스트로 확장될 때 개별 호스트 단위의 예측 및 최적화 방식이 초래할 계산·운영 비용 문제, 호스트 간 상호의존성 및 집단 최적화 미고려로 인한 한계점, 그리고 동기 처리의 병목 현상을 중점적으로 살펴봅니다.

## 1. 핵심 한계 요약
*   호스트별 독립적인 순차 처리 방식은 대규모 플릿에서 심각한 병목 현상을 초래합니다.
*   플릿 단위의 자원 최적화(통합, 상호의존성) 부재로 전체 시스템의 자원 효율성이 저하됩니다.
*   GCP API 연동 시 동기 블로킹 호출로 인해 대규모 인스턴스 동기화 및 리사이징 작업이 지연됩니다.
*   매번 전체 시계열 데이터를 읽어 재적합하는 방식은 데이터가 많아질수록 성능 저하를 야기합니다.

## 2. 상세 분석

### 2.1 호스트별 예측 및 최적화의 비효율
*   **[심각도: 높음]** 현재 MetricLens의 예측(`backend/app/core/forecaster.py`) 및 최적화(`backend/app/core/optimizer.py`) 로직은 개별 호스트의 시계열 데이터에 대해 독립적으로 수행됩니다.
*   **근거(인용)**:
    *   `docs/02_architecture_volume.md`의 "Core 알고리즘" 설명에서 `forecaster`와 `optimizer`가 순수 함수로서 개별 호스트의 시계열을 처리함을 명시합니다.
    *   `docs/05_sequence_diagram.md`는 메트릭 적재, 예측, 리사이징 권장 흐름이 모두 특정 `host_id`에 대해 단일 트랜잭션으로 진행됨을 보여줍니다.
    *   `backend/app/services.py`의 `AnalysisService.forecast` 및 `AnalysisService.recommend` 메서드는 `host_id`를 인자로 받아 해당 호스트의 샘플만 조회(`self.metrics.list_by_host(host_id)`)하여 코어 로직에 전달합니다.
    *   `docs/12_limitations.md`의 "2.1 "정수계획법 / fleet" 명칭과 구현의 괴리" 섹션에서 `optimizer.py`가 "호스트별로 독립 수행되는 1차원 완전탐색"이며 "다중 호스트 빈패킹·통합(consolidation)·co-location·마이그레이션이 없다"고 지적합니다.
*   **왜 문제**: 수천 대의 호스트를 관리하는 플릿에서는 각 호스트에 대해 순차적으로 (또는 단순히 병렬로) 이러한 작업을 수행하는 것만으로도 막대한 시간과 컴퓨팅 자원이 소요됩니다. 이는 대규모 플릿의 빠른 상태 변화에 대한 적시 대응을 어렵게 만듭니다.
*   **어떤 조건에서 드러나는가**: 모니터링 대상 호스트의 수가 수백 대 이상으로 증가하거나, 플릿 전체의 예측/최적화 작업이 빈번하게 요구될 때 (예: 시간당 1회) 계산 시간이 급격히 증가하여 병목이 됩니다.

### 2.2 플릿 수준 상호작용 및 전역 최적화 부재
*   **[심각도: 높음]** 현재 시스템은 호스트들을 상호 독립적인 존재로 간주하며, 호스트 간의 워크로드 상호의존성, 공유 자원, 오토스케일링 그룹 등을 전혀 고려하지 않습니다.
*   **근거(인용)**: `docs/12_limitations.md`의 "2.1 "정수계획법 / fleet" 명칭과 구현의 괴리" 섹션에서 "다중 호스트 빈패킹·통합(consolidation)·co-location·마이그레이션이 없다. 진정한 fleet-level 정수계획이 아니다"라고 명확히 기술합니다. `optimizer.py`의 `recommend_resize` 함수는 오직 단일 `current_vcpu`, `current_memory_mb` 값을 받아 최적화를 수행합니다.
*   **왜 문제**: 실제 프로덕션 환경에서는 마이크로서비스 아키텍처나 분산 시스템에서 호스트 간의 복잡한 의존 관계가 존재합니다. 예를 들어, 웹 서버와 데이터베이스 서버는 밀접한 관계를 가지므로, 웹 서버의 리사이징이 데이터베이스 서버의 부하에 영향을 미칠 수 있습니다. 개별 최적화는 이러한 의존성을 무시하여 전체 시스템 관점에서는 비효율적인 자원 배분, 심지어는 성능 저하를 야기할 수 있습니다.
*   **어떤 조건에서 드러나는가**: 서비스 간 강한 의존성이 있는 분산 시스템, 또는 자원 집약적인 백엔드 서비스와 이를 사용하는 다수의 프론트엔드 서비스가 혼재하는 환경에서 개별 최적화의 한계가 명확하게 드러납니다.

### 2.3 GCP 통합 시 동기 블로킹 호출로 인한 병목
*   **[심각도: 중간]** GCP Cloud Monitoring에서 메트릭을 가져오거나 Compute Engine 인스턴스를 리사이징하는 작업이 동기적으로 (혹은 `asyncio.to_thread`를 사용하지만 여전히 순차적으로) 처리되어, 대규모 플릿에서는 I/O 병목이 발생할 수 있습니다.
*   **근거(인용)**:
    *   `backend/app/services_gcp.py`의 `GcpSyncService.sync` 메서드는 `gcp.list_instances`로 인스턴스 목록을 가져온 후, 각 인스턴스에 대해 `gcp.fetch_cpu_series` 및 `gcp.fetch_mem_series`를 호출하는 루프를 포함합니다. 이들 함수는 `google.cloud` 클라이언트 라이브러리를 사용하며, 코어 로직은 동기적으로 작동합니다.
    *   `GcpSyncService.real_resize`는 `gcp.resize_instance`를 호출하는데, 이는 VM을 `stop → setMachineType → start`하는 일련의 동기 블로킹 작업입니다.
*   **왜 문제**: Python의 `asyncio` 기반 FastAPI 애플리케이션에서 동기 블로킹 I/O 작업은 이벤트 루프를 막아 다른 비동기 요청 처리를 지연시킵니다. 수백~수천 대의 인스턴스에 대해 메트릭을 동기화하거나 리사이징 명령을 순차적으로 보내는 것은 전체 플릿 관리 작업의 응답 시간을 매우 길게 만들 것입니다.
*   **어떤 조건에서 드러나는가**: `GcpSyncService.sync`가 실행될 때 GCP 인스턴스의 수가 많아지거나, 여러 호스트에 대해 `real_resize`를 동시에 수행하려고 할 때 병목이 나타납니다.

### 2.4 데이터 처리 방식의 비효율성 (재학습/재계산)
*   **[심각도: 중간]** 현재 예측 모델은 매번 예측을 수행할 때마다 대상 호스트의 전체 시계열 데이터를 조회하여 모델을 재적합하는 방식(`AnalysisService.forecast` 참조)을 사용합니다.
*   **근거(인용)**: `backend/app/services.py`의 `AnalysisService.forecast`는 `self.metrics.list_by_host(host_id)`를 통해 모든 과거 데이터를 가져온 후 `forecaster.forecast(series, ...)`를 호출합니다. `docs/12_limitations.md`의 "5.3 단일 인스턴스·SQLite" 섹션에서 "매 예측이 전체 시리즈를 재적합하나(O(n)) N이 작아 당장은 무해하다"고 언급합니다. (여기서 N은 시계열 길이).
*   **왜 문제**: 시계열 데이터의 길이가 길어지거나 예측/최적화 요청이 빈번해지면, 매번 전체 데이터를 읽고 재계산하는 오버헤드가 커집니다. 이는 특히 시계열 데이터가 수백 또는 수천 개의 샘플을 포함할 경우 예측 응답 시간을 증가시킵니다.
*   **어떤 조건에서 드러나는가**: 데이터 보존 기간이 길어져 시계열 데이터의 `series_length`가 증가하거나, 예측 빈도가 높아질 때 (예: 1시간보다 짧은 간격) 계산 비용이 문제가 됩니다.

## 3. 암묵적 가정과 그 취약성
*   **플릿 규모는 작다**: 현재 설계는 MetricLens가 관리할 호스트의 수가 수십 대 정도의 작은 규모일 것이라는 가정을 하고 있습니다. 이 가정이 깨지면 (수백~수천 대의 호스트) 시스템은 급격히 느려지거나 응답 불능 상태에 빠질 것입니다.
*   **호스트는 상호 독립적이다**: 각 호스트의 자원 사용 패턴이 다른 호스트에 영향을 미치지 않으며, 각 호스트를 개별적으로 최적화해도 플릿 전체의 효율성이나 서비스 가용성에 문제가 없다는 가정을 합니다. 복잡한 분산 시스템에서는 이러한 가정이 쉽게 깨집니다.
*   **실시간에 가까운 응답 시간은 불필요하다**: 예측 및 최적화 과정에서 발생하는 수 초에서 수 분의 지연이 허용된다는 가정을 합니다. 수천 대의 호스트에 대한 작업을 수행할 경우 이 지연은 훨씬 길어져 실시간 운영에는 부적합하게 됩니다.
*   **자원 제약이 있는 환경은 아니다**: `forecaster`와 `optimizer`가 Python 표준 라이브러리 기반으로 GPU 없이 CPU만으로 작동하는 "경량" 모델임에도 불구하고, 대규모 플릿에 대한 `O(N * series_length)` 복잡성의 계산이 충분한 자원 내에서 감당될 것이라는 가정을 합니다.

## 4. 파급 효과
*   **성능 저하 및 확장성 한계**: 대규모 플릿에서는 예측 및 최적화 작업이 지연되거나 타임아웃되어 시스템의 실용성이 떨어집니다. Cloud Run의 `max-instances=10` 설정(`cloudbuild.yaml`)도 처리량을 제한하여, 수천 대 호스트의 동시 요청 처리에는 부족합니다.
*   **비용 비효율성**: 호스트별 독립 최적화는 플릿 전체 관점에서 자원 통합이나 효율적인 배치 기회를 놓쳐, 불필요한 자원 낭비 및 비용 증가로 이어질 수 있습니다.
*   **운영 복잡도 증가**: 자동화된 플릿 관리가 어려워지고, 운영자가 개별 호스트를 수동으로 관리해야 하는 상황이 발생할 수 있습니다.
*   **SLO/SLA 위협**: 상호의존적인 서비스의 경우 개별 호스트 최적화가 전체 서비스의 SLO/SLA를 위협할 수 있습니다. 예를 들어, 한 서비스가 다운사이징되어 병목이 되면 다른 서비스 전체의 성능에 악영향을 줍니다.

## 5. 개선 제안
1.  **플릿 단위 비동기 배치 처리 도입**: `AnalysisService`에 플릿 전체 호스트에 대한 예측 및 최적화를 비동기적으로 스케줄링하고 결과를 취합하는 `batch_forecast` 또는 `batch_recommend`와 같은 메서드를 추가합니다. 이는 Celery와 같은 태스크 큐 시스템과 연동하여 분산 처리를 고려할 수 있습니다.
2.  **플릿 수준 최적화 알고리즘 도입**: `backend/app/core/optimizer.py`에 단일 호스트가 아닌 전체 플릿을 대상으로 하는 통합 최적화(예: 빈패킹, 다차원 배낭 문제 변형) 로직을 추가하여 자원 통합 및 배치 효율성을 높입니다.
3.  **GCP API 연동 비동기화 및 병렬 처리**: `GcpSyncService` 내 GCP 클라이언트 호출 시 `asyncio.gather`를 활용하여 `gcp.fetch_cpu_series`, `gcp.resize_instance` 등 잠재적 블로킹 I/O 작업을 병렬로 실행합니다. 또한, GCP API 호출 시 `asyncio.to_thread`를 사용하여 블로킹 함수를 스레드 풀에서 실행함으로써 이벤트 루프를 해방합니다.
4.  **시계열 데이터 증분 처리 및 캐싱**: 예측 및 최적화 시 매번 모든 과거 데이터를 로드하는 대신, 변경된 부분만 업데이트하거나, 이미 계산된 중간 결과를 캐싱하여 `forecast` 함수의 입력 `series` 길이를 줄이는 방안을 모색합니다. Redis와 같은 인메모리 데이터스토어를 활용할 수 있습니다.

## 6. 타 관점과의 연결
*   **#01 forecasting-model (예측 모델의 표현력 한계)**: 대규모 플릿에서는 단순한 단변량 시계열 모델 대신 다변량 모델이나 계층적 모델이 필요할 수 있으며, 이는 계산 복잡도를 높여 확장성 문제를 악화시킬 수 있습니다.
*   **#03 optimization (정수계획 최적화의 한계)**: 현재의 호스트별 독립 최적화는 플릿 규모에서 가장 큰 한계점이며, 플릿 단위 최적화로 전환될 경우 정수계획 문제의 복잡도가 기하급수적으로 증가할 수 있습니다.
*   **#08 realtime-ops (실시간성·운영의 한계)**: 예측/최적화 작업의 처리 지연은 실시간 운영 환경에서의 빠른 의사결정과 자동 리사이징을 어렵게 하여 운영 응답성에 직접적인 영향을 미칩니다.
*   **#11 architecture-debt (아키텍처·기술부채의 한계)**: 확장성 문제를 해결하기 위한 아키텍처 변경(분산 처리, 메시지 큐 등)은 상당한 기술 부채를 초래할 수 있습니다.
