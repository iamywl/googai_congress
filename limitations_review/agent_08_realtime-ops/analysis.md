## 0. 관점 메모
이 렌즈에서는 MetricLens AI의 서버 리소스 리사이징 기능이 실제 운영 환경(특히 실시간 서비스)에 미치는 영향에 집중하여, 다운타임, 리사이징 빈번도(flapping), 안정화(cooldown) 메커니즘 부재, 그리고 모니터링 주기로 인한 반응성 한계를 분석한다.

## 1. 핵심 한계 요약
*   실제 VM 리사이징 과정에서 발생하는 명시적 다운타임은 SLO 무중단 보장과 직접적으로 상충한다.
*   리사이징 결정 주기가 시간 단위 모니터링에 의존하여 실시간 부하 변화에 대한 적시성 확보가 어렵다.
*   리사이징 진동(flapping) 방지 및 안정화를 위한 쿨다운/히스테리시스 메커니즘이 부재하다.
*   상태유지(stateful) 서비스에 대한 리사이징 고려 사항이 없으며, 이로 인한 데이터 손실 또는 서비스 중단 위험이 크다.

## 2. 상세 분석

*   **한계 1: 실제 VM 리사이징 시 발생하는 명시적 다운타임**
    *   **심각도**: 치명
    *   **무엇이**: `backend/app/integrations/gcp.py`의 `resize_instance` 함수는 `client.stop()`, `client.set_machine_type()`, `client.start()` API 호출을 순차적으로 수행한다. 이는 VM을 중지하고, 머신 타입을 변경한 후 다시 시작하는 과정으로, 서비스 중단을 필연적으로 수반한다.
    *   **근거**:
        *   `backend/app/integrations/gcp.py`: "resize_instance(project: str, zone: str, name: str, machine_type: str) -> None: """Change a VM's machine type: stop → setMachineType → start. Blocking.""" "
        *   `docs/10_uml_models.md`의 "12. 타이밍 다이어그램": "실제 리사이즈는 stop → setMachineType → start 순으로 진행되며, 다운타임은 t1~t4 구간이다."
        *   `docs/12_limitations.md`의 "4. 실제 GCP 통합 (데모 수준) - 리사이즈 다운타임": "`stop → setMachineType → start` ...로 다운타임이 발생한다. 라이브 리사이즈가 아니라 SLO 무중단 보장과 상충한다."
    *   **왜 문제**: Cloud Run과 같은 서버리스 환경에서 무중단 배포(`docs/02_architecture_volume.md`의 "무중단 배포")를 지향하는 시스템이, 핵심 기능인 VM 리사이징에서는 명시적인 다운타임을 발생시킨다. 이는 현대 클라우드 운영의 기본 원칙인 고가용성(High Availability) 및 무중단 서비스 제공 요구사항과 근본적으로 충돌하며, "SLO를 보장하는 최소 자원 구성"(`CLAUDE.md`, `README.md`)이라는 핵심 가치를 훼손한다.
    *   **어떤 조건에서 드러나는가**: 모든 실제 GCP VM 리사이징 작업에서 발생하며, 특히 미션 크리티컬한 서비스나 상태유지(stateful) 서비스를 대상으로 할 때 치명적인 영향을 미친다.

*   **한계 2: 모니터링 및 반응 주기의 지연과 적시성 부족**
    *   **심각도**: 높음
    *   **무엇이**: 시스템이 메트릭 데이터를 시간 단위(`settings.sample_interval_minutes: int = 60`)로 수집하고, 예측 모델 또한 이 시간 단위 데이터를 기반으로 작동한다. Cloud Monitoring 통합에서도 시간 단위(`align_minutes: int = 60`)로 데이터를 가져온다.
    *   **근거**:
        *   `backend/app/config.py`: `sample_interval_minutes: int = 60`
        *   `backend/app/services.py`: `horizon_steps = max(1, round(horizon_minutes / settings.sample_interval_minutes))`
        *   `backend/app/integrations/gcp.py`: `fetch_cpu_series` 함수에서 `align_minutes: int = 60`
        *   `docs/12_limitations.md`의 "4. 실제 GCP 통합 (데모 수준) - 수집 해상도": "Cloud Monitoring 시간 정렬로 시간당 1포인트, 전체 일주기 확보에 ~24h, period=24 백테스트엔 2일 이상 필요 → 콜드스타트 공백."
    *   **왜 문제**: 서버 부하는 초, 분 단위로 급격하게 변동할 수 있다. 시간 단위의 모니터링 및 예측 주기는 이러한 단기적인 부하 스파이크나 급격한 변화를 적시에 감지하고 대응하기 어렵게 만든다. "SLO를 보장"한다는 목표는 실시간에 가까운 반응성을 요구하지만, 현재 시스템은 최소 1시간의 지연을 내재하고 있다. 이는 부하 급증 시 서비스 품질 저하 또는 장애로 이어질 수 있다.
    *   **어떤 조건에서 드러나는가**: 트래픽 패턴이 불규칙하거나, 예측 불가능한 스파이크 부하가 발생하는 서비스에서 문제가 심각하게 드러난다. 짧은 시간에 자원 증설이 필요한 경우, 1시간의 지연은 너무 길다.

*   **한계 3: 리사이징 진동(flapping) 방지 및 안정화(cooldown) 메커니즘 부재**
    *   **심각도**: 중간
    *   **무엇이**: `optimizer.py`의 `recommend_resize` 함수는 현재 상태와 예측 피크를 기반으로 즉각적인 최적화를 수행한다. 리사이징이 적용된 후 새로운 예측이 발생하면 다시 최적화를 진행하며, 리사이징 주기를 제어하거나 반복적인 리사이징을 방지하는 쿨다운 기간, 히스테리시스 로직이 없다.
    *   **근거**: `optimizer.py` 및 관련 서비스 코드(`AnalysisService`)에서 리사이징 후 특정 기간 동안 재조정을 금지하거나, 일정 범위 내 변화는 무시하는 로직이 발견되지 않음. `docs/12_limitations.md`의 "2.4 비용·운영 현실 미반영": "리사이즈 다운타임·쿨다운·플래핑(oscillation) 미고려."
    *   **왜 문제**: 예측 오차, 미세한 부하 변동, 또는 예측 모델의 특성으로 인해 시스템이 지속적으로 자원 규모를 변경하라고 권고할 수 있다. 이는 VM 재시작(다운타임)을 빈번하게 유발하여 서비스 가용성을 저해하고, 운영 비용(특히 stop/start 시 발생하는 부수 비용)을 증가시킬 수 있다.
    *   **어떤 조건에서 드러나는가**: 예측 모델의 민감도가 높거나, 실제 부하가 예측 구간 경계에서 미세하게 변동하는 경우, 또는 시스템이 안정화되기 전에 재차 리사이징 권고가 발생하는 경우.

*   **한계 4: 상태유지 서비스에 대한 고려 부재**
    *   **심각도**: 치명
    *   **무엇이**: 현재 리사이징 방식은 VM 중지(`stop`)를 포함하며, 이는 서비스가 일시적으로 오프라인 상태가 된다는 의미이다. 만약 해당 VM이 데이터베이스, 캐시 서버, 메시지 큐 등 상태유지(stateful) 서비스를 실행하고 있다면, 강제 종료 및 재시작 과정에서 데이터 손실, 데이터 불일치, 또는 복구에 오랜 시간이 소요되는 문제가 발생할 수 있다.
    *   **근거**: `backend/app/integrations/gcp.py`의 `resize_instance` 함수는 VM을 무조건 중지시킨다. `docs/10_uml_models.md`의 "7. 상태 머신 다이어그램"에서 `Stopping` 상태로 진입하는 것을 명시. `CLAUDE.md`의 "MetricLens AI" 정의는 "경량 시계열 모델로 서버 부하(CPU·메모리·네트워크)를 예측"으로, 특정 서비스 타입(stateful/stateless)을 구분하지 않는다.
    *   **왜 문제**: 상태유지 서비스는 일반적으로 콜드 스타트(cold start)에 더 많은 시간이 소요되며, 복구 로직이 복잡하다. 단순한 `stop/start`는 데이터 무결성을 훼손하거나 서비스의 장기적인 안정성을 심각하게 위협할 수 있다.
    *   **어떤 조건에서 드러나는가**: 데이터베이스, 메시지 브로커, 분산 캐시 등 상태를 저장하고 유지해야 하는 서비스에 MetricLens AI를 적용할 경우.

## 3. 암묵적 가정과 그 취약성
*   **가정**: 모든 모니터링 대상 서버는 상태를 가지지 않는(stateless) 서비스이거나, 서비스 중단(downtime)이 허용되는 배치(batch)성 워크로드이다.
    *   **취약성**: 실제 운영 환경에서는 데이터베이스, 캐시, 세션 서버 등 상태를 유지해야 하는 서비스가 다수 존재한다. 이러한 서비스에 현재 리사이징 방식을 적용하면 데이터 손실, 서비스 중단, 데이터 불일치 등의 치명적인 문제가 발생한다.
*   **가정**: 1시간 단위의 메트릭 수집 및 예측이 부하 변화를 감지하고 최적의 자원을 제공하기에 충분하다.
    *   **취약성**: 서비스 부하는 매우 짧은 시간(수분~수초) 내에 급격히 변동할 수 있다. 1시간 간격의 데이터는 이러한 단기적 변화를 놓치고, 예측이 항상 현실에 뒤처지는(lagging) 문제를 야기하여 SLO를 위반할 수 있다.
*   **가정**: 예측 모델의 출력(리사이징 권고)은 즉시 적용 가능하며, 운영 안정성에 대한 추가적인 보호 메커니즘이 필요 없다.
    *   **취약성**: 예측 모델은 항상 오류를 포함하며, 이 오류나 외부 환경 변화로 인해 불필요하거나 반복적인 리사이징 권고가 발생할 수 있다. 이를 제어하는 장치가 없으면 시스템이 불안정해지고 운영 비용이 증가한다.

## 4. 파급 효과
*   **서비스 가용성 저하**: 강제적인 다운타임과 빈번한 리사이징으로 인해 서비스 중단 시간이 증가하여 사용자 경험과 비즈니스 연속성에 부정적인 영향을 미친다.
*   **데이터 무결성 문제**: 상태유지 서비스에 대한 고려 없는 리사이징은 데이터 손실 및 무결성 훼손으로 이어질 수 있으며, 이는 복구에 막대한 시간과 비용을 소모하게 한다.
*   **운영 비용 증가**: 잦은 VM 재시작은 인프라 오버헤드, 로그 증가, 모니터링 복잡도 증가 등 운영 비용을 증가시킨다.
*   **SLO 미달성**: "SLO를 보장하는 최소 자원 구성"이라는 핵심 목표가 실시간 운영 환경의 제약으로 인해 달성되지 못하거나, 오히려 서비스 안정성을 해칠 수 있다.
*   **시스템 신뢰도 하락**: 예측/최적화 시스템이 실제 운영 환경에서 문제를 야기하면, 시스템 전반에 대한 신뢰도가 하락하여 도입 및 확산에 큰 장애가 될 수 있다.

## 5. 개선 제안
*   **무중단 리사이징 방식 도입**: GCP Live Migration, Kubernetes Pod Graceful Shutdown, 또는 미리 프로비저닝된 인스턴스 스왑 등 다운타임이 없는(또는 최소화된) 리사이징 기술 통합을 고려한다. `integrations/gcp.py`의 `resize_instance` 로직을 재검토하고, `compute_v1.InstancesClient.update` 등 다른 API 호출 가능성을 탐색한다.
*   **메트릭 수집 및 예측 주기 세분화**: 최소 5분~1분 단위의 메트릭 수집 및 예측 주기를 도입하여 부하 변화에 대한 반응성을 높인다. `backend/app/config.py`의 `sample_interval_minutes`와 `gcp_sync_hours` 설정을 조정하고, Cloud Monitoring API 호출 시 `align_minutes`를 변경한다.
*   **리사이징 제어 로직 추가**:
    *   **쿨다운(Cooldown) 기간**: 리사이징 적용 후 일정 시간(예: 30분~1시간) 동안 추가 리사이징 권고를 무시하는 로직을 서비스 계층(`AnalysisService`)에 구현한다.
    *   **히스테리시스(Hysteresis)**: 리소스 사용률이 특정 임계값 범위 내에서 미세하게 변동할 경우 리사이징을 하지 않도록 허용 오차를 둔다.
    *   `docs/12_limitations.md`의 "2.4 비용·운영 현실 미반영"에서 언급된 "플래핑 억제(히스테리시스) 추가"를 구현한다.
*   **상태유지 서비스 워크로드 명시 및 처리 전략**:
    *   호스트 등록 시 `stateless` 또는 `stateful` 속성을 추가하고, `stateful` 서비스에 대해서는 자동 리사이징을 기본적으로 비활성화하거나, 별도의 수동 승인 워크플로우를 제공한다.
    *   데이터베이스 등의 서비스는 리사이징 시 복제본 교체(failover) 또는 롤링 업데이트 방식의 리사이징을 유도한다.

## 6. 타 관점과의 연결
*   **#03 optimization (정수계획 최적화의 한계)**: 현재의 독립적인 호스트별 최적화 방식은 쿨다운 및 히스테리시스 로직 부재와 결합하여 전체 플릿 관점의 비효율적인 리사이징 진동을 악화시킬 수 있다. 플릿 단위 최적화는 이러한 운영 문제를 완화하는 데 도움이 될 수 있다.
*   **#04 slo-model (SLO 모델링의 한계)**: 현재의 SLO 보장은 리사이징으로 인한 다운타임이나 실시간성 부족을 고려하지 않는다. 실제 SLO를 보장하려면 운영 상의 제약(다운타임, 복구 시간)이 모델에 반영되어야 한다.
*   **#14 explainability-ux (설명가능성·운영 UX의 한계)**: 운영자가 리사이징 결정을 신뢰하려면, 리사이징 시 발생할 수 있는 다운타임, 서비스 영향, 복구 시간 등에 대한 명확한 정보와 위험 고지가 필요하다. 현재의 "화이트박스 설명가능성"이 이러한 운영적 측면을 충분히 다루고 있지 못하다.
