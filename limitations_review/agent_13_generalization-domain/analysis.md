## 0. 관점 메모
이 분석 렌즈는 MetricLens AI가 현재 다루는 CPU 및 메모리라는 핵심 지표를 넘어, 다른 종류의 성능 메트릭과 다양한 워크로드 패턴에 얼마나 유연하게 적용될 수 있는지, 그리고 GCP라는 특정 클라우드 환경 외 다른 인프라(컨테이너, 서버리스, 엣지)로의 이식성에 대한 한계점을 집중적으로 살펴봅니다.

## 1. 핵심 한계 요약
*   **지원 메트릭 제한**: CPU, 메모리, 기본 네트워크 I/O 외 디스크 IOPS, GPU, 동시 접속 등 다양한 성능 지표를 고려하지 않습니다.
*   **워크로드 패턴 가정**: 주기성 및 특정 패턴(대화형, 배치)을 강하게 가정하며, 비주기적이거나 예측 불가능한 워크로드(예: 이벤트성, 콜드스타트)에 대한 대응력이 부족합니다.
*   **인프라 종속성**: GCP Compute Engine의 VM 리소스 모델에 최적화되어 있으며, 컨테이너, 서버리스 또는 엣지 컴퓨팅 환경으로의 직접적인 이식 및 최적화가 어렵습니다.

## 2. 상세 분석

### 2.1 지원 메트릭의 제한성
*   **심각도**: 높음
*   **무엇이**: 현재 MetricLens AI의 예측 및 최적화 대상 메트릭은 CPU 사용률(`cpu_pct`), 메모리 사용률(`mem_pct`), 네트워크 수신/송신 트래픽(`net_in_kbps`, `net_out_kbps`)으로 제한됩니다. 디스크 IOPS, GPU 사용률, 동시 접속자 수, 큐 대기 시간과 같은 다른 중요한 성능 메트릭들은 전혀 고려되지 않습니다.
*   **근거(인용)**:
    *   `backend/app/schemas.py` 파일의 `MetricKind` enum은 `CPU`, `MEM`, `NET_IN`, `NET_OUT`만을 정의하고 있습니다.
    *   `backend/app/models.py` 파일의 `Metric` 모델 또한 이 네 가지 필드만 포함합니다.
    *   `docs/01_functional_spec.md`의 "메트릭별 예측" 항목은 이 네 가지 메트릭만을 나열합니다.
    *   `backend/app/core/optimizer.py`의 `recommend_resize` 함수는 `predicted_peak_cpu_pct`와 `predicted_peak_mem_pct`만을 입력으로 받습니다.
*   **왜 문제**: 특정 애플리케이션(예: 데이터베이스, 파일 서버)은 디스크 I/O에, 머신러닝 워크로드는 GPU에, 웹 서비스는 동시 접속자 수에 따라 성능 병목이 발생할 수 있습니다. 이러한 메트릭이 리소스 최적화에 반영되지 않으면, MetricLens AI의 권장 사항이 실제 서비스 성능에 부정적인 영향을 미치거나 불필요한 과도한 리소스 할당을 초래할 수 있습니다.
*   **어떤 조건에서 드러나는가**: 디스크 I/O나 GPU, 동시 접속자 수가 핵심 병목 지점인 서비스에 MetricLens AI를 적용할 경우, 잘못된 리사이징 권장으로 서비스 지연 또는 장애가 발생할 수 있습니다.

### 2.2 워크로드 패턴의 일반화 한계
*   **심각도**: 중간
*   **무엇이**: MetricLens AI의 시계열 예측 모델은 `period` (주기) 개념을 사용하며, 주로 `interactive` (일주기), `batch` (버스트), `steady` (평탄)와 같은 특정 통계적 패턴을 가진 워크로드를 가정합니다. 비주기적이거나 급격한 변동성을 가진 이벤트성 워크로드, 또는 콜드스타트 지연이 중요한 서버리스 함수와 같은 환경에는 예측 및 최적화 모델이 부적합할 수 있습니다. `p95` 피크 로직은 단발성 스파이크를 무시하여 안정성을 높이지만, 중요한 이벤트성 트래픽을 과소평가할 위험도 내포합니다.
*   **근거(인용)**:
    *   `backend/app/core/forecaster.py`의 `forecast` 함수는 `period` 인자를 필수로 받으며, `_seasonal_indices`와 `_decompose` 함수가 주기성 기반 분해를 수행합니다.
    *   `backend/app/core/workload.py` 및 `docs/09_workload_modeling.md`에서 묘사된 6가지 워크로드 아키타입은 모두 주기성, 버스트성 등 특정 패턴을 명시적으로 부여합니다. `_DIURNAL` 상수는 일주기성을 명확히 반영합니다.
    *   `backend/app/core/optimizer.py`의 `peak` 함수는 `percentile`을 이용하여 단발성 스파이크의 과도한 리소스 할당을 방지합니다.
*   **왜 문제**: 주기가 없거나 매우 불규칙한 서비스(예: 캠페인성 트래픽, 비정기 대규모 데이터 처리)에 대한 예측 정확도가 현저히 떨어질 수 있습니다. 또한, 콜드스타트 워크로드의 경우, 리소스 할당의 최소값이 아닌 '활성화 시간' 자체가 중요한데, 이는 현재 모델에서 전혀 다뤄지지 않습니다.
*   **어떤 조건에서 드러나는가**: 예측 불가능한 외부 이벤트로 인해 트래픽이 급증하는 경우, 또는 새로 배포된 서버리스 함수가 처음 호출될 때 발생하는 콜드스타트 문제를 MetricLens AI가 해결하지 못하고, 오히려 비효율적인 리소스 권장을 할 수 있습니다.

### 2.3 인프라 이식성 및 기술 스택 종속성
*   **심각도**: 중간
*   **무엇이**: MetricLens AI는 GCP Cloud Run 및 Compute Engine 환경에 대한 통합에 중점을 둡니다. 특히 리사이징 권장 사항은 GCP의 `machine_types` 카탈로그를 활용하며, 실제 리사이징은 Compute Engine API를 통해 이루어집니다. 이는 컨테이너 오케스트레이션 플랫폼(Kubernetes), 다른 클라우드 제공업체, 또는 온프레미스 베어메탈/VMware 환경으로의 직접적인 이식 및 활용에 제약을 가합니다.
*   **근거(인용)**:
    *   `CLAUDE.md` 및 `README.md`는 "GCP 네이티브: Cloud Run + Cloud Build" 배포를 명확히 합니다.
    *   `backend/app/core/machine_types.py`는 `GCE predefined machine-type catalogue`을 포함하며, `nearest_fit` 함수는 GCP 인스턴스에 맞춰 리소스를 '스냅'합니다.
    *   `backend/app/integrations/gcp.py`는 `google.cloud.compute_v1` 및 `google.cloud.monitoring_v3` 클라이언트를 사용하여 GCP 리소스와 직접 상호작용합니다.
    *   `frontend/src/api.js`의 `SAMPLE_MACHINE_TYPES` 또한 GCP 머신 타입을 모방합니다.
*   **왜 문제**: "에어갭/온프레 자립형"이라는 차별점(`docs/08_competitive_analysis.md`)을 강조하지만, 실제 구현은 GCP에 강하게 종속되어 있습니다. 코어 로직 자체는 순수 Python으로 구현되었지만, 리사이징 권장의 실질적인 적용 부분은 GCP에 고정되어 있어 다른 인프라 환경에서 사용하기 위해서는 상당한 통합 작업이 필요합니다. 이는 클라우드 벤더 락인 문제를 야기하며, "멀티클라우드/온프레/VM·베어메탈 무관"이라는 주장을 약화시킵니다.
*   **어떤 조건에서 드러나는가**: AWS EKS 또는 온프레미스 Kubernetes 클러스터에서 MetricLens AI를 사용하여 파드(Pod) 또는 노드 리소스를 최적화하려 할 때, `vcpu_count`와 `memory_mb` 기반의 최적화 모델과 GCP 머신 타입 스냅 로직이 적절하지 않아 통합이 불가능하거나 비효율적입니다.

## 3. 암묵적 가정과 그 취약성
MetricLens AI는 CPU와 메모리 사용률이 서버 성능의 주요 결정 요인이며, 워크로드 패턴이 주기적이거나 예측 가능한 형태를 따른다고 암묵적으로 가정합니다. 또한, 모든 최적화 결정이 VM 수준의 vCPU 및 메모리 할당으로 충분하다고 가정합니다. 이러한 가정은 디스크 I/O가 병목인 스토리지 서버, GPU가 핵심인 딥러닝 워크로드, 또는 트랜잭션 수가 성능을 좌우하는 데이터베이스 서버와 같이 특정 전문화된 환경에서는 깨질 수 있습니다. 인프라의 `provider` 필드를 `demo` 또는 `gce`로 명시하는 것은 다른 제공업체를 기본적으로 배제하는 가정입니다.

## 4. 파급 효과
*   **최적화의 불완전성**: 핵심 지표가 누락된 경우, MetricLens AI가 제시하는 최적화 권장 사항이 실제 병목을 해결하지 못해 리소스 낭비를 지속하거나, 오히려 다른 자원의 병목을 심화시켜 서비스 품질 저하를 초래할 수 있습니다.
*   **적용 가능 도메인 축소**: 특정 워크로드 패턴에 대한 의존성은 MetricLens AI의 적용 가능 도메인을 축소시킵니다. 금융 거래 시스템의 급작스러운 이벤트성 트래픽이나, 새로운 데이터 파이프라인의 비정기적 대규모 처리 작업 등에는 효과적인 대응이 어렵습니다.
*   **재활용성 및 확장성 저해**: GCP Compute Engine에 강하게 종속된 구현은 다른 클라우드 환경(AWS, Azure)이나 컨테이너/서버리스 플랫폼(Kubernetes, Lambda)에서의 재활용성을 크게 제한하며, 이는 MetricLens AI의 시장 확장성을 저해하는 요인이 됩니다.

## 5. 개선 제안
1.  **플러그인 방식의 메트릭 통합**: CPU/메모리 외 디스크 IOPS, GPU 활용률, 동시 접속자 수 등을 플러그인 형태로 추가할 수 있는 유연한 메트릭 수집 및 분석 프레임워크를 설계합니다. (심각도: 높음)
2.  **워크로드 패턴 분류 확장**: 비주기적이거나 이벤트 중심의 워크로드를 식별하고, 이에 특화된 예측 모델(예: 극단값 이론, 변동성 모델링) 또는 규칙 기반 최적화 전략을 도입합니다. (심각도: 중간)
3.  **추상화된 인프라 모델**: GCP `machine_types`와 같은 구체적인 인프라 종속성을 추상화 계층 뒤로 숨기고, 다양한 인프라 제공업체에 대한 어댑터 패턴을 구현하여 이식성을 높입니다. (심각도: 중간)
4.  **컨테이너/서버리스 특화 최적화**: vCPU/메모리 외 컨테이너의 파드 밀도, 레플리카 수, 서버리스 함수의 동시성 및 콜드스타트 시간 등을 고려한 최적화 모델을 추가합니다. (심각도: 낮음)

## 6. 타 관점과의 연결
*   **#01 예측 모델의 표현력 한계**: 비선형성 및 다중 계절성 문제를 해결하여 비주기 서비스의 예측 정확도를 높이는 것이 이 관점과 긴밀하게 연결됩니다.
*   **#03 정수계획 최적화의 한계**: 현재 호스트별 독립 최적화를 넘어서 다차원 메트릭과 복합 인프라 환경을 고려한 진정한 플릿 최적화(빈패킹, co-location)로 확장해야 합니다.
*   **#05 데이터 대표성의 한계**: 합성 워크로드 외 실제 트레이스 데이터로 다양한 메트릭 및 워크로드 패턴에 대한 모델을 검증해야 합니다.
*   **#12 GCP 통합의 한계**: 메모리 미측정 프록시나 Cloud Monitoring의 수집 해상도와 같은 문제는 메트릭 확장 시 데이터 품질 문제로 이어질 수 있습니다.
