## 0. 관점 메모
이 분석 렌즈는 MetricLens AI 프로젝트가 시계열 모델 훈련 및 평가에 사용하는 데이터의 대표성 문제를 집중적으로 검토한다. 특히, 합성 워크로드 데이터에 대한 의존성, 실제 트레이스 데이터의 부재, 이로 인해 발생하는 모델의 일반화 능력 한계 및 도메인 시프트에 대한 취약성을 분석한다.

## 1. 핵심 한계 요약
*   **합성 데이터에 대한 전적인 의존**: 모든 모델 학습 및 평가는 자체 생성한 합성 워크로드 데이터(`backend/app/core/workload.py`)에 전적으로 의존한다.
*   **실측 트레이스 데이터의 부재**: 평가 과정에 실제 데이터센터 트레이스(`docs/09_workload_modeling.md`에서 언급된 Azure/Alibaba 트레이스)가 직접 사용되지 않아, 모델의 외적 타당성이 제한된다.
*   **순환 구조로 인한 자기 참조적 평가**: 데이터 생성 과정(`_AR_RHO`, `_DAY_AMP` 등)에 모델이 유리하도록 인위적인 통계적 특성이 주입되어 있으며, 이 데이터로 모델을 평가하는 **순환 구조**(`docs/12_limitations.md` §3.1)는 평가의 객관성을 저해한다.
*   **도메인 시프트 취약성**: 합성 데이터가 실제 환경의 모든 복잡성과 미묘한 분포 변화를 반영하지 못하므로, 실제 운영 환경에 배포 시 예측 성능이 급격히 저하될 위험이 있다.

## 2. 상세 분석

*   **한계 1: 모델 학습 및 평가의 합성 데이터 의존**
    *   **심각도**: 치명
    *   **무엇이**: MetricLens AI의 모든 시계열 예측 모델 학습 및 평가는 `backend/app/core/workload.py`에서 정의된 6가지 합성 워크로드 아키타입(예: `interactive_web`, `batch_etl`)과 이를 기반으로 `_SEED_DAYS = 14`일간 생성된 데이터에 전적으로 의존한다. `docs/11_model_evaluation.md` 및 `backend/tests/test_evaluation.py`의 모든 평가 케이스가 이 합성 데이터를 사용한다.
    *   **근거(인용)**:
        *   `docs/09_workload_modeling.md`: "본 문서는 MetricLens의 데모/시드 데이터와 시험 시나리오가 임의로 작성된 것이 아니라, **공개된 대규모 데이터센터 트레이스의 실측 통계에 근거(calibration)** 하여 구성되었음을 기술한다."
        *   `docs/12_limitations.md` §3.1: "자기 생성 데이터로 자기 모델 평가 - **현상**: 평가는 [`workload.py`](../backend/app/core/workload.py)가 만든 합성 데이터에서만 수행된다. ... **영향**: 모델이 유리하도록 데이터 생성 과정에 정렬된 **순환 구조**다."
        *   `backend/app/core/workload.py`: `generate` 함수는 `_AR_RHO`, `_DAY_AMP`, `_ANOMALY_THRESH`와 같은 고정된 확률적 파라미터로 데이터를 생성한다.
    *   **왜 문제**: 합성 데이터는 정의된 통계적 특성(AR(1) 자기상관, 일주기성, 버스트성 등)을 재현하도록 설계되었으나, 실제 시스템에서 발생하는 예측 불가능한 노이즈, 희귀 이벤트, 비선형적인 상호작용 등을 완전히 포착하기 어렵다. 이는 모델이 실제 환경에서 검증되지 않았음을 의미한다.
    *   **어떤 조건에서 드러나는가**: 모델이 실제 운영 환경(프로덕션 트래픽, 사용자 행동 변화, 새로운 애플리케이션 배포 등)에 배포될 때, 합성 데이터에서 보장되었던 예측 정확도(`docs/11_model_evaluation.md`의 MAPE/MASE 지표)가 크게 저하될 수 있다.

*   **한계 2: 실측 트레이스 데이터의 평가 부재**
    *   **심각도**: 높음
    *   **무엇이**: `docs/09_workload_modeling.md`에서 Azure Resource Central, Barroso & Hölzle, Alibaba 2018 트레이스 데이터를 근거 자료로 언급하지만, 이 실제 트레이스 데이터가 모델의 학습 또는 **평가(홀드아웃 검증)**에 직접적으로 사용되지 않는다.
    *   **근거(인용)**:
        *   `docs/09_workload_modeling.md`: "세 출처가 공통적으로 가리키는 사실: **실제 서버는 만성적으로 저활용**이며... MetricLens의 시드는 이 세 축을 모두 재현한다." (재현에 그침)
        *   `docs/12_limitations.md` §3.1: "모델이 유리하도록 데이터 생성 과정에 정렬된 **순환 구조**다. `docs/11`의 우수 지표(MASE<1, DM p<0.05)는 이 자기참조 위에 있어 외적 타당성이 제한된다."
        *   `docs/12_limitations.md` §3.1 **보완**: "Azure/Alibaba 등 **실제 트레이스 1개 이상으로 홀드아웃 재평가**."
    *   **왜 문제**: 실제 트레이스 데이터는 합성 데이터로는 모방하기 어려운 복잡한 패턴, 이상치, 장기적 추세 변화 등을 포함한다. 이러한 데이터를 통한 외부 검증이 없으면, 모델이 특정 합성 데이터에 과적합(overfitting)되었을 가능성을 배제할 수 없다. "통계적으로 대표성"을 주장하지만, 이를 독립적인 실측 데이터로 검증하지 않는 것은 모델의 신뢰성을 낮춘다.
    *   **어떤 조건에서 드러나는가**: 모델이 합성 데이터와 통계적 특성이 다른 실제 워크로드에 적용될 때, 또는 기존의 합성 워크로드 패턴에서 크게 벗어나는 새로운 워크로드 패턴이 등장할 때.

*   **한계 3: 결정론적 이상치 주입의 현실성 부족**
    *   **심각도**: 중간
    *   **무엇이**: 합성 워크로드 생성 시 `_ANOMALY_THRESH = 0.99`를 사용하여 약 1%의 시간대에 +12~34% 스파이크를 결정론적으로 주입한다(`backend/app/core/workload.py`).
    *   **근거(인용)**: `backend/app/core/workload.py` `_ANOMALY_THRESH = 0.99`, `_unit(seed + "a", i) > _ANOMALY_THRESH`, `cpu += 12.0 + 22.0 * _unit(seed + "x", i)`
    *   **왜 문제**: 실제 시스템의 이상치나 장애 이벤트는 특정 확률 분포를 따르거나 예측 불가능한 형태로 발생할 수 있다. 결정론적으로 주입된 이상치는 모델이 특정 패턴의 이상치에만 반응하도록 훈련시키고, 예상치 못한 새로운 종류의 이상치에 대해서는 강건성을 보이지 못할 수 있다.
    *   **어떤 조건에서 드러나는가**: 실제 운영 환경에서 합성 데이터의 결정론적 패턴과 다른 형태나 규모의 이상치가 발생할 때, 모델이 이를 정확하게 예측하거나 처리하지 못할 수 있다.

## 3. 암묵적 가정과 그 취약성
MetricLens AI는 합성 데이터가 "실제 데이터센터 트레이스의 실측 통계에 근거(calibration)"되었으므로, 이 데이터로 학습 및 평가된 모델이 실제 운영 환경에서도 유사한 성능을 보일 것이라는 암묵적인 가정을 하고 있다. 특히 `docs/09_workload_modeling.md`에 명시된 "통계적 검증(실측 정렬)" 및 "모델 평가 — '제대로 동작하는가' (통계적 당위성)" 섹션은 이 가정을 뒷받침하려 한다.

그러나 이러한 가정은 다음과 같은 조건에서 취약하다:
*   **미반영된 복잡성**: 합성 데이터가 재현하지 못한 실제 워크로드의 미묘한 특징(예: 다중 계절성, 복잡한 비선형 관계, 외부 요인에 의한 급격한 변화)이 존재할 경우. `docs/12_limitations.md` §1.3 "단일 주기 가정", §1.4 "예측구간의 정규·등분산 가정"에서 이러한 한계가 일부 인정되고 있다.
*   **데이터 분포의 시프트(Domain Shift)**: 시간이 지남에 따라 실제 워크로드의 분포가 합성 데이터의 분포와 달라질 경우. 이는 새로운 서비스 도입, 사용자 규모 변화, 계절적 이벤트 등으로 인해 발생할 수 있다.
*   **희귀 이벤트/블랙 스완**: 합성 데이터에 포함되지 않은 매우 드물지만 치명적인 이벤트(예: 대규모 DDoS 공격, 하드웨어 장애)에 대한 모델의 대응 능력이 검증되지 않았다.

## 4. 파급 효과
데이터 대표성 한계는 시스템 전반에 걸쳐 심각한 파급 효과를 초래할 수 있다.
*   **예측 정확도 저하**: 모델이 합성 데이터에 과적합되어 실제 환경에서 `docs/11_model_evaluation.md`에 보고된 MAPE 및 MASE 지표를 달성하지 못할 수 있다. 이는 리사이징 권고의 신뢰도를 떨어뜨린다.
*   **비효율적인 자원 최적화**: 부정확한 예측은 `backend/app/core/optimizer.py`의 `recommend_resize` 함수가 실제 필요한 자원보다 과도하거나 부족한 자원을 권고하게 만든다. 이는 불필요한 비용 발생(오버 프로비저닝) 또는 SLO 위반(언더 프로비저닝)으로 이어진다.
*   **운영자의 신뢰 상실**: 모델의 권고가 실제 운영 결과와 자주 불일치할 경우, 인프라 관리자는 MetricLens AI 시스템에 대한 신뢰를 잃고 수동 개입을 선호하게 될 것이다.
*   **규제 감사/설명가능성 문제**: "화이트박스 설명가능성"(`CLAUDE.md` §7, `README.md` "차별점")을 강조하지만, 근본 데이터의 대표성이 부족하면 설명의 근거가 약해져 외부 감사 시 모델의 타당성을 입증하기 어려울 수 있다.
*   **오류 및 버그 발생 가능성**: 합성 데이터에서는 발생하지 않았던 실제 데이터의 특정 패턴이나 에지 케이스가 운영 환경에서 모델의 예기치 않은 동작이나 오류를 유발할 수 있다.

## 5. 개선 제안
1.  **실제 워크로드 트레이스를 통한 홀드아웃 평가 도입 (우선순위: 높음)**: `docs/12_limitations.md` §3.1에서 스스로 제안했듯이, Azure/Alibaba 등 **공개된 실제 데이터센터 트레이스 중 최소 1개 이상을 홀드아웃 데이터셋으로 확보하여 모델을 재평가**해야 한다. 이를 통해 합성 데이터로만 검증된 "순환 구조"를 해소하고 모델의 외적 타당성을 확보한다.
2.  **다양한 실제 워크로드 패턴 반영 (우선순위: 중간)**: 합성 워크로드 생성 시, 현재 6가지 아키타입 외에 더 많은 실제 시나리오(예: 계절적 프로모션, 개발-운영 주기, 예상치 못한 트래픽 급증)를 반영할 수 있는 메커니즘을 추가하여 데이터셋의 다양성을 높인다.
3.  **데이터 증강 및 시뮬레이션 개선 (우선순위: 중간)**: 실제 데이터 트레이스를 직접 사용하기 어렵다면, 현재의 결정론적 잡음 주입 방식(`backend/app/core/workload.py`의 `_noise`, `_ANOMALY_THRESH`)을 개선하여 실제 워크로드의 통계적 특성(예: 멱법칙 분포, 버스트성, long-tail 이벤트)을 더 잘 모방하는 고급 데이터 증강 기법(예: GAN, Diffusion Model)을 탐색한다.
4.  **모델 모니터링 및 재학습 파이프라인 구축 (우선순위: 낮음)**: 실제 환경에 배포된 후 모델의 성능 저하(도메인 시프트)를 감지할 수 있는 모니터링 시스템을 구축하고, 실제 운영 데이터를 주기적으로 학습에 반영하는 재학습(retraining) 파이프라인을 고려한다.

## 6. 타 관점과의 연결
*   **예측 모델의 표현력 한계 (#01)**: 합성 데이터가 실제 워크로드의 복잡성을 완전히 담아내지 못하는 경우, 현재의 `STL_HOLTWINTERS` 모델이 실제 데이터의 비선형성, 다중 계절성, 다변량 상호작용 등을 포착하지 못할 수 있다.
*   **평가 엄밀성의 한계 (#06)**: 합성 데이터에 대한 평가 결과는 실제 환경에서의 모델 성능을 과대평가할 수 있으며, 1-스텝 백테스트만으로는 다중 스텝 예측의 불확실성을 충분히 검증하지 못한다. 홀드아웃 데이터셋 도입 시 평가 지표 및 방법론을 재검토해야 한다.
*   **GCP 통합의 한계 (#12)**: 현재 GCP 연동을 통해 수집되는 실제 CPU 메트릭(`gcp.fetch_cpu_series`)은 제한된 해상도(시간당 1포인트)와 메모리 프록시 값(`gcp.fetch_mem_series`가 비어 있을 경우 CPU 기반 추정)에 의존한다. 이러한 실제 메트릭 데이터가 모델 훈련에 사용될 경우, 모델이 불완전하거나 부정확한 정보에 기반하여 학습될 수 있다.
*   **설명가능성·운영 UX의 한계 (#14)**: 데이터 대표성 문제가 심각해 예측과 권고의 신뢰도가 떨어지면, 아무리 모델의 추론 과정이 화이트박스라 할지라도 운영자가 그 설명을 받아들이고 시스템을 신뢰하기 어렵다.
